# Audit of PLAN.md (pre-M0)

Six-dimension audit with every falsifiable claim independently verified against
MLX/mlx-lm source, Qwen3 configs + technical report, and the mini-sglang repo.
The PLAN and this scaffold already fold in the fixes below.

## Verdict

The spine is sound: decode is memory-bandwidth-bound at batch=1, quantization is
the right central lever, "% of peak bandwidth" is the honest figure of merit,
and gating custom kernels on profiling is the best decision in the plan. The
risks are (a) correctness omissions that block a correct M0 and (b) a
thesis/novelty problem — most of M0–M2 already ships in `mlx-lm`.

## Tier 1 — would break M0 (now fixed in scaffold)

1. **Qwen3 QK-Norm omitted.** Per-head RMSNorm on Q and K over `head_dim`,
   before RoPE — the biggest Qwen3-vs-Qwen2 delta. Also: Qwen3 dropped QKV bias;
   `head_dim`=128 is decoupled (≠ 1024/16); 0.6B ties embeddings. → `model.py`.
2. **No output path.** Plan ended at token IDs: no UTF-8-safe detokenizer, no
   EOS/EOT, no string stop-sequences, no ChatML template. → `detokenize.py`,
   `generate.py`.

## Tier 2 — factual errors (now corrected in PLAN)

3. **SDPA does not handle quantized KV in one call** — `mx.fast.scaled_dot_product_attention`
   is fp16/bf16/fp32 only; quantized KV is a separate `quantized_matmul`×2 path.
4. **"growing → quantized → rotating" doesn't compose** — `RotatingKVCache.to_quantized()`
   raises `NotImplementedError`; they are alternatives.
5. **"Unified memory = structural advantage"** is true vs discrete GPUs but not
   vs the `mlx-lm` / `llama.cpp`-Metal baselines (same platform).
6. **async_eval hides per-step dispatch + host↔device sync**, not "graph-build
   overhead" (`mx.compile` caches the trace).

## Tier 3 — methodology gaps (now corrected in §6 + bench/)

7. **Figure of merit counted only weights** — must add KV(context) + lm_head;
   KV dominates at long context (Qwen3-4B: ~10× weights at 128k).
8. **No measurement rigor** — warmup, variance/CI, thermal throttling
   (laptop M3 Max), tok/s/W. → `BenchConfig`, `bench/decode.py`.
9. **Quantized footprint** must include per-group scales+biases (~12.5% over
   packed at 4-bit/g64, ~4.5 effective bits/weight).
10. **Bandwidth denominator unpinned** — M3 Max is 300 OR 400 GB/s; record SKU.
11. **No acceptance criteria** — added M0 tolerance + M4 fairness checklist.

## Tier 4 — strategy

12. **Thesis concentrates novelty in M3**, which §7 itself hedges (M0–M2 already
    in `mlx-lm`). → pedagogy made a gated M4 deliverable; thesis reframed to
    "quantify mlx-lm's headroom and test whether fusion closes it."
13. **Top unlisted risk: `mlx-lm` may already run near peak bandwidth** → M2
    baseline-first go/no-go.
14. **batch=1 is a structural ceiling**, not just a deferral — now owned in §2.
15. **Quantized KV is memory-for-speed (~0.5× fp16)** — measure, don't assume.
16. **Internal contradiction** on "the" top lever (async_eval vs quantization).

## Tier 5 — completeness (folded into roadmap)

Mixed-precision quant policy (keep embed/lm_head higher); max-context + YaRN;
RoPE θ=1e6; sampler beyond greedy; perplexity eval harness; rotating eviction
semantics (sink+window); weight-loading specifics (single vs sharded, bf16→mlx,
lazy read-on-eval — NOT mmap); seeding/config/logging/CI; group_size
divisibility.

## Confirmed strengths (kept)

Bandwidth-bound thesis + quantization-as-lever; % peak bandwidth metric; M3
gated on M2 profiling with pure-MLX fallbacks; ruthless honest scope; accurate
mini-sglang characterization; mostly-correct MLX API reading.
